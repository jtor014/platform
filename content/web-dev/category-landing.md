# Web Development

Build full-stack web applications by directing AI agents. Start with a static website, end with a production-grade multi-user system with monitoring, backups, and handover documentation. Every project ships to a real URL.

Web Development is the first category because it produces the most tangible results the fastest — you can show someone a working website after your first project. The skills you learn here (planning, directing, verifying, deploying) transfer directly to every other category.

## How this is different from coding tutorials

Coding tutorials teach you to write code. This teaches you to direct AI to write it — and to catch when AI gets it wrong. You won't learn JavaScript syntax. You will learn to read a pull request, spot scope creep in an AI-generated PRD, verify a payment integration against real Stripe documentation, and break your own access controls before a user does.

The output isn't "I understand React." It's "I shipped six deployed products and I can explain every decision I made."

## The progression

Six projects. Each repeats the same loop with one new capability — and one shift in what's expected of you as a Director.

| Project | What You Build | How Your Role Changes |
|---|---|---|
| 1. The Static Site | A professional business website for a landscaping company. Deployed to a live URL | You learn the loop: branch, direct, verify, merge. Everything is new |
| 2. The Dynamic App | A full-stack facts app. React, FastAPI, PostgreSQL, Docker, CI, Gemini reviewer | You learn to verify systems — databases, APIs, containers — not just pages |
| 3. The Interactive App | Forms, validation, email. You plan it yourself from a client brief | You write the PRD. Planning judgment becomes the new skill |
| 4. The Integrated App | Third-party API integration (Stripe, Google Maps, or similar) | You verify AI's work against external documentation. Hallucination detection |
| 5. The Multi-User App | Auth, roles, permissions. Different users see different things | You think adversarially. Can you break your own access controls? |
| 6. The Live System | Monitoring, backups, security, handover documentation | You make it production-grade. Could someone else keep this running? |

## The arcs

Projects 1-2 are shared foundations — everyone does them. Projects 3-5 are contextual variations. You choose a business story (an "arc") that provides the client briefs, scope changes, and domain-specific challenges for those three projects. The Framework skills are the same across all arcs — the domain is what makes each one unique.

| Arc | Domain | Signature Challenge |
|---|---|---|
| A: Service Business | Landscaping/trades | Progressive client relationship, full lifecycle |
| B: Online Store | Handmade products | Financial accuracy, Stripe verification |
| C: Content Platform | Community publication | Anti-scope vigilance, non-technical users |
| D: Event Platform | Workshops/markets | Date/time precision, offline-first |
| E: Community Directory | Business directory | Search specification, adversarial trust |

You don't need to choose your arc until after Project 2. The [bridge exercises](/web-dev/bridge) help you decide.
