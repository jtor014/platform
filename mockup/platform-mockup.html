How this works

You direct. AI builds. You verify. Nothing ships without your approval.

The roles

👤

You — the Director

Decide what gets built. Review the work. Verify it's correct. Sign off on every merge.

⚡

Claude Code

AI developer. Writes code, creates files, opens PRs. Fast and capable, but makes mistakes.

🔍

Gemini reviewer

Independently reviews every PR. Catches what Claude missed: security issues, edge cases.

The loop

Every project follows the same cycle. You'll do this dozens of times until it's muscle memory.

1

Plan with AI

Give AI a client brief and a template. It produces a PRD. You review: scope creep? Vague criteria? Approve or direct changes.

2

Build ticket by ticket

You say "Start Ticket N." Claude branches, implements, tests, opens a PR. Two AI reviews run automatically.

3

Verify by using it

Open the product. Test like a real user. Try to break it. Check the design. Check the data.

4

Deploy and reflect

Ship to a real URL. Verify in production. Run a retro. Lessons feed into the next project.

Checkpoint: After project 1

Can you spot problems, not just follow instructions?

Exercise 1: Anti-scope detection

Three PRD excerpts below. The client said: "I need a website that shows my services, some photos, and how to contact me."

Excerpt A: Homepage, services list, photo gallery, contact section with phone and email.

Excerpt B: Homepage, services, portfolio, contact form with database, newsletter signup, blog, analytics, live chat widget.

Excerpt C: Single-page site with services, work photos, about section, contact details, SEO meta tags.

Excerpt B has five features the client never asked for: database contact form, newsletter, blog, analytics, live chat. This is exactly what AI does — assumes a business site needs all of these.

Exercise 2: Design verification

The spec says: card border-radius 8px, padding 24px, heading font-size 20px. The implementation shows: border-radius 12px, padding 16px, font-size 18px.

How many discrepancies? What specifically is wrong?

Three discrepancies: border-radius (12 vs 8), padding (16 vs 24), font-size (18 vs 20). "It looks roughly right" isn't good enough. You check specific values.

Exercise 3: Review reading

Gemini flags: "The email address is in plain HTML, exposing it to harvesting bots. Consider using JavaScript obfuscation or a contact form."

Is this a real issue? What do you do?

The concern is valid but the fix is wrong. A contact form was explicitly anti-scoped. JavaScript obfuscation adds complexity for a minor risk. Acknowledge in the retro as a known tradeoff, don't implement a fix.

PRD: GreenScape Landscaping

Product requirements for the static business website.

1. Executive summary

Build a professional static website for GreenScape Landscaping — a small landscaping business in Melbourne. The site showcases services, displays past work, and provides contact information. Single page, responsive, deployed to a live URL.

2. Problem statement

The client has no online presence. Potential customers searching for landscapers in their area can't find GreenScape. The client is losing business to competitors who have websites.

3. Scope — what we're building

Hero section

Full-width hero with business name, tagline ("Transform your outdoor space"), and a call-to-action button linking to the contact section.

Acceptance criteria:

Hero is the first thing visible without scrolling on desktop (1440px) and mobile (375px)
CTA button scrolls smoothly to the contact section
Business name renders in the correct font (Playfair Display) at ≥48px desktop / ≥32px mobile

Services section

Six service cards displaying: Garden Design, Lawn Care, Paving & Paths, Retaining Walls, Irrigation, Tree Services. Each card has an icon, title, and 2-sentence description.

6 cards displayed in 3-column grid on desktop, 1-column on mobile
Each card matches the design spec: #FFFFFF background, 1px #E5E5DC border, 8px border-radius

4. Anti-scope

Do NOT build any of the following:

No contact form (phone and email links only) · No database · No blog · No CMS or admin panel · No user accounts · No analytics · No newsletter signup · No live chat · No e-commerce or payment · No booking system · No multi-page navigation (single page only)

5. Success metrics

Lighthouse performance score above 90
All sections match the design spec (verified with DevTools)
Live at a public URL with SSL
Loads in under 2 seconds on 4G

Step-by-step guide

Your first project. Follow this guide ticket by ticket.

Before you start

Make sure you've completed the Shared setup and Web Dev setup. You need: VS Code, Git, GitHub CLI, Claude Code, Node.js, and a Netlify account.

Starting your first ticket

1

Create the project

Open your terminal and create the project directory:

$ mkdir greenscape && cd greenscape
$ git init
$ gh repo create greenscape --public --source=. --remote=origin
$ code .
2

Start Claude Code

In the VS Code integrated terminal:

$ claude

Claude Code starts. You're now directing.

3

Create CLAUDE.md

Tell Claude:

"Create a file called CLAUDE.md with exactly the following content. Then commit it to main."

Paste the full CLAUDE.md from the CLAUDE.md page.

4

Say "Start Ticket 1"

Claude creates a feature branch, initialises the project, sets up the Makefile, and opens a PR.

"Start Ticket 1."

What to verify after Ticket 1

After Claude reports back with CI status and review, verify yourself:

Open localhost:5173 — do you see the placeholder page?
Open VS Code — do you see the file tree on the left?
Run make dev — does the server start?

If everything checks out, tell Claude: "Merge it."

The loop from here

Every subsequent ticket follows the same pattern:

→

You say "Start Ticket N"

Claude branches, implements, tests, opens a PR, self-reviews, waits for Gemini.

→

Claude reports back

CI status, self-review, Gemini review, PR size, recommendation.

→

You verify

Open the browser. Check against the design spec. Use DevTools. Try to break it.

→

You decide

"Merge it" or "Fix those."

Common issue — Ticket 3: Claude may use different colours than the design spec. Open DevTools (F12), inspect the hero section, and check the actual hex values against the spec. If they don't match, tell Claude: "The hero background is [wrong colour]. It should be #2D5016. Fix it."
